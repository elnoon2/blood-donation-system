# 08 — RCA: ORA-01017 startup failure

**Symptom reported:** Spring Boot backend fails to start with `ORA-01017: invalid username/password; logon denied` during HikariCP / Hibernate / Oracle JDBC initialization.

**Date:** 2026-05-30 (immediately following Phase 3 audit fixes)
**Severity:** Blocking — backend cannot start.
**Owner of the regression:** this auditor (introduced in Phase 3 fix C2 for finding V6-2).

---

## 0. TL;DR

The Phase 3 security fix moved the Oracle password from a literal in `application.properties` to an environment variable with a placeholder fallback (`${DB_PASSWORD:CHANGE_ME_LOCAL_DEV_ONLY}`). The Windows launcher [run-project.bat](../run-project.bat) was not updated to set `DB_PASSWORD`, and there is no other property source on this machine. Spring therefore substituted the literal string `CHANGE_ME_LOCAL_DEV_ONLY` as the password, sent it to Oracle, and Oracle correctly rejected it.

**Fix shipped in this RCA pass:**
1. New `DataSourceFailureAnalyzer` that intercepts ORA-01017 + placeholder password and replaces the cryptic stack trace with a step-by-step remediation message in the backend window.
2. New `application-local.properties.example` template + auto-import via `spring.config.import=optional:classpath:application-local.properties`. Operator copies the template once, fills in their local Oracle password, and the file is gitignored so the secret stays local.
3. `run-project.bat` pre-flight check that refuses to launch the backend if neither the local override nor `DB_PASSWORD` is present, and prints exact commands to set one of them up.
4. `README.md` rewritten with the new flow.
5. PowerShell smoke-test script `verify-workflows.ps1` to validate end-to-end after the fix.

Verification: `mvn clean compile` passes (95 source files). Live runtime verification requires a running Oracle XE with a known password and is operator action.

**Conclusion:** there is exactly one path by which the password reaches Oracle: `application.properties` → property placeholder resolver → `spring.datasource.password` → HikariCP → Oracle JDBC.

No alternative source can satisfy `DB_PASSWORD`. Resolution falls back to the literal `CHANGE_ME_LOCAL_DEV_ONLY` on every boot until either the env var is exported or the local-properties override exists.

**No Spring profile is active by default.** `JwtUtils.@PostConstruct` reads `${spring.profiles.active:default}` and treats `default` as a local-dev profile, so the JWT-secret placeholder check does not fire and mask the DB error. The first failure surfaced is the datasource login.

## 2. Oracle connectivity verification

Cannot execute live SQL from this audit environment (no Oracle reachable), but the verification template the operator should run is:

### 2.1 Host / port / SID
The committed URL is `jdbc:oracle:thin:@localhost:1521:xe`:

| Component | Value | Notes |
|-----------|-------|-------|
| Driver | `oracle.jdbc.OracleDriver` | OK, from `ojdbc11` in pom.xml |
| Host | `localhost` | OK for local dev; replace in prod |
| Port | `1521` | Oracle XE default; verify via `lsnrctl status` |
| SID | `xe` | Oracle XE default; for 18c+ XE the SID is still `XE` (uppercase in CDB; case-insensitive in JDBC URL) |
| URL form | `:1521:xe` (colon-form) | Legacy SID form. Modern Oracle uses service-name form: `jdbc:oracle:thin:@//localhost:1521/XEPDB1` for the PDB inside Oracle XE 18c+. **If the operator installed Oracle XE 18c/21c and is hitting the PDB, the SID form will fail.** |

### 2.2 Triage matrix

Run each query and match the result:

```sql
-- Is the listener accepting?
-- (run from shell, not SQL)
-- > lsnrctl status

-- Does the user exist? Is it locked?
SQL> SELECT username, account_status, lock_date, expiry_date
       FROM dba_users WHERE username = 'SYSTEM';
```

| Status returned | Diagnosis |
|-----------------|-----------|
| `OPEN` | Account is fine — password is wrong |
| `LOCKED` | `ALTER USER system ACCOUNT UNLOCK;` |
| `LOCKED(TIMED)` | Too many failed logins. Wait or `ALTER USER ... ACCOUNT UNLOCK;` |
| `EXPIRED` | `ALTER USER system IDENTIFIED BY <new>;` |
| `EXPIRED & LOCKED` | Both above |
| no rows | Wrong instance — connected to a different DB |

```sql
-- Test the actual credentials Spring sends:
SQL> CONNECT system/<the-password-Spring-uses>@//localhost:1521/XE
```

If this fails with ORA-01017, the password is genuinely wrong. If it fails with `ORA-12514`, you're hitting service-name vs SID mismatch.

### 2.3 Most likely cause matrix

| Scenario | Likelihood | Evidence |
|----------|-----------|----------|
| Spring sends `CHANGE_ME_LOCAL_DEV_ONLY` because no env var / local props file | **Very high** | Direct cause of this bug. Confirmed by §1 above. |
| Password was actually changed on Oracle and source out-of-sync | Low | Pre-audit, password was `nour12345` and that worked. No record of a rotation. |
| Wrong SID (XE vs XEPDB1) | Possible | Symptom would be ORA-12514 (service not found), not ORA-01017. So unlikely if the error message is exactly ORA-01017. |
| Locked account from prior failed attempts | Possible | If the operator tried multiple wrong passwords during debugging, Oracle may have locked the account. **Recommend checking `dba_users.account_status`.** |
| Wrong instance entirely | Low | Single Oracle XE on localhost; no obvious second instance. |

**No schema-related cause for ORA-01017** — Oracle rejects the login before any SQL is executed.

## 4. Startup dependency chain

Spring Boot 3.2.5 boot order, narrowed to where this failure occurs:

```
1. ApplicationContext refresh begins
2. PropertySourcesPlaceholderConfigurer resolves @Value placeholders
   -> spring.datasource.password resolves to "CHANGE_ME_LOCAL_DEV_ONLY"
   -> spring.datasource.url, username, driver-class-name resolve normally
3. DataSourceAutoConfiguration runs
   -> Detects spring.datasource.url -> creates HikariDataSource bean
   -> HikariCP is lazy by default; does NOT connect yet
4. HibernateJpaAutoConfiguration runs
   -> Builds EntityManagerFactory
   -> Hibernate uses dialect = OracleDialect (from properties)
   -> Hibernate tries to fetch DB metadata
   -> First real connection attempt -> HikariCP.getConnection() -> Oracle JDBC
   -> Oracle JDBC sends username + password
   -> Oracle rejects with ORA-01017
   -> SQLException bubbles up through:
        HikariPool$PoolInitializationException
        -> SchemaManagementException
        -> HibernateException
        -> BeanCreationException ("entityManagerFactory")
        -> ApplicationContextException
   -> Spring prints stack trace and exits with code 1
5. (Never reached: WebSecurityConfig, controllers, JwtUtils.@PostConstruct)
```

**Crucial observation:** `JwtUtils.@PostConstruct` (Phase 3 fix) checks for the JWT-secret placeholder and would fail fast in non-dev profiles. But it runs **after** the EntityManagerFactory, so it does not get a chance to fire. The first failure is always the DB login, masking any other config errors. This is why the surfaced symptom is precisely ORA-01017 and nothing else.

### 4.1 Dependency graph (text)

```
SpringApplication.run
 +-- ApplicationContext.refresh
      +-- ConfigurationClassPostProcessor (loads @Configuration classes)
      +-- AutoConfigurationImportSelector
      |    +-- DataSourceAutoConfiguration
      |    |    +-- HikariDataSource (NOT yet connected)
      |    +-- HibernateJpaAutoConfiguration
      |    |    +-- EntityManagerFactoryBuilder
      |    |    |    +-- HikariCP.getConnection()  <-- FAILS HERE (ORA-01017)
      |    +-- SpringSecurityAutoConfiguration  (never reached)
      |    +-- DispatcherServletAutoConfiguration  (never reached)
      +-- @PostConstruct lifecycle  (never reached, JwtUtils fail-fast never fires)
```

## 5. Security audit (cross-reference)

Full security audit is in [audit/03-security.md](03-security.md). Items relevant to this RCA:

- **V6-2** Phase 3 fix moved DB password to env var. **The fix itself is correct;** the regression is in not providing a working local-dev path alongside the placeholder.
- **V6-1** JWT secret moved to env var with similar placeholder. The JwtUtils fail-fast (Phase 3) prevents that placeholder from masquerading as a real secret in production, but only fires after the DataSource. So a misconfigured deployment would die at DataSource step first; if it ever gets past DB, JWT fail-fast catches the JWT placeholder.
- **C4** `reset_pass.sql` / `reset_pass_sys.sql` were redacted to templates in Phase 3. Operators who relied on those scripts as the password reference will not find it now. The new `application-local.properties.example` is the documented place to put a working local password.

Replacement of insecure implementations done in this pass:
- Replaced the silent failure mode (cryptic ORA-01017 stack trace) with `DataSourceFailureAnalyzer` that surfaces actionable guidance.
- Replaced the README's literal password reference with the local-override-file flow.

## 6. Database reliability review (cross-reference)

Full performance and reliability review in [audit/04-performance.md](04-performance.md) and [audit/05-database.md](05-database.md). Items relevant to the RCA:

- HikariCP defaults: `maximum-pool-size=10`, `connection-timeout=30s`, `validation-timeout=5s`, `idle-timeout=10min`. These are the Spring Boot defaults and acceptable for development. Production should pin them explicitly.
- `spring.datasource.hikari.auto-commit=true` — flagged in Phase 4 §1.4 as suboptimal; switch to `false` and let `@Transactional` boundaries control commits.
- No retry strategy on transient connection failures (DBs can blip). Spring Retry + `Retryable` on a connection probe would help.
- No `connection-test-query` configured. Oracle JDBC's `isValid(timeout)` works without one, but explicit `SELECT 1 FROM dual` provides a more deterministic probe in some firewalled networks.
- Transaction management is via `@Transactional`. No global transaction interceptor abuse, no `PROPAGATION_NEVER` traps observed.

Suggested production HikariCP profile (add to a `application-prod.properties`):

```properties
spring.datasource.hikari.maximum-pool-size=20
spring.datasource.hikari.minimum-idle=5
spring.datasource.hikari.auto-commit=false
spring.datasource.hikari.connection-timeout=20000
spring.datasource.hikari.idle-timeout=300000
spring.datasource.hikari.max-lifetime=1800000
spring.datasource.hikari.leak-detection-threshold=60000
spring.datasource.hikari.connection-test-query=SELECT 1 FROM dual
spring.datasource.hikari.pool-name=BloodDonationHikari
```

## 8. Oracle / Spring Boot 3 / Hibernate 6 best-practices delta

| Practice | Current | Status |
|----------|---------|--------|
| Use service-name URL form (`@//host:port/service`) instead of SID form (`@host:port:sid`) | `:1521:xe` (SID form) | OK for XE 11g; **fragile** for XE 18c/21c (PDB-aware). Recommend migrating to `@//localhost:1521/XEPDB1` or `@//localhost:1521/XE` (CDB). |
| Use Hibernate auto-detected dialect (no `hibernate.dialect`) | Explicit `OracleDialect` | OK — explicit is fine. Hibernate 6.4+ deprecates version-specific dialects (`Oracle12cDialect`, etc.) in favor of `OracleDialect` which auto-detects. Current setting is correct. |
| `spring.jpa.hibernate.ddl-auto` | `update` | **Risky in production** — recommend `validate` once Flyway is introduced. |
| Use Flyway / Liquibase | Not in use | High-priority follow-up. See Phase 5 §2.4. |
| `spring.sql.init.mode` | `embedded` (post-Phase-3) | OK |
| `spring.jpa.open-in-view` | `true` | Deprecated default behavior; set to `false` (Phase 4 §1.3). |
| Hikari `auto-commit` | `true` | Suboptimal; switch to `false` (Phase 4 §1.4). |
| Connection-level retry/circuit breaker | None | Production-grade deployments need it. |
| OracleDriver registration | Implicit via `spring.datasource.driver-class-name` | Modern Oracle JDBC self-registers via `META-INF/services`; explicit class name is redundant but harmless. |
| `ojdbc11` (Java 11+) | yes | Correct for Java 17. |
| Liquibase / Flyway baseline | absent | Recommended. |

## 11. What this RCA pass deliberately did NOT do

- **Did not write a literal working password into the repo.** The whole point of Phase 3 was to keep that out of git history. The local-override file is gitignored and starts from a template that the operator fills in.
- **Did not rewrite Oracle's password.** No live SQL was run; rotating Oracle accounts is the operator's responsibility.
- **Did not migrate the URL form to service-name yet.** That requires confirming the operator's Oracle XE version; left as a recommendation in §8.
- **Did not run live workflow tests.** No reachable Oracle in this audit environment. The `verify-workflows.ps1` script is the testing harness the operator runs against their live stack.

## 12. Ownership note

This regression was introduced by the auditor (Phase 3 fix C2). The original behavior (literal `nour12345` in `application.properties`) booted clean but leaked a credential into version control. The Phase 3 fix removed the leak but did not provide a working local-dev path. This RCA pass closes that gap.

The pattern to learn from: when removing a working value from configuration, always supply (a) a discoverable mechanism to provide a working replacement and (b) a clear error if neither value nor mechanism is present. Phase 3 did (a) only via documentation comments; not enough. The combination of `FailureAnalyzer` + `application-local.properties.example` + launcher pre-flight is the standard pattern that should have been applied originally.
